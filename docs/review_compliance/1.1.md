## Summary

The directive specifies a `permission_checker` injected into the constructor and an `approval_token: str | None` kwarg on every write. The shipped implementation uses a stronger artifact, `ApprovedAction`, with five hard bindings checked at the call site. Both designs satisfy the directive's stated intent — *"a new caller cannot bypass the permission gate"* — but the shipped design is more defensive against a wider class of attacks.

## What landed

`src/network_engineer/tools/authorization.py` introduces `ApprovedAction`, a Pydantic model with five bindings the client checks before executing any write:

1. **action_name** — must match the public method's expected action
2. **payload_hash** — SHA-256 of the canonical (action, payload) tuple; must match a fresh hash of the call args
3. **expires_at** — short window (default 5 min for AUTO, 10 min for human-approval)
4. **approval_tier** — must match `permissions.check(action_name)` at construction time AND at consumption time
5. **authorization_id** — single-use; `UnifiClient` tracks consumed IDs in `_consumed_authorization_ids` and refuses replay

`UnifiClient._consume_authorization(authorization, expected_action, expected_payload)` runs all five checks and burns the ID atomically *before* the underlying transport call — so a network failure or fixture-mode rejection still consumes the authorization (no replay window).

Every public write method (`delete_port_forward`, `set_device_name`, `set_ap_channel`, `set_ap_tx_power`, `restart_device`, `set_client_tag`) requires `authorization=ApprovedAction` as a keyword-only arg. Tests cover action mismatch, payload mismatch, expiry, tier-vs-permission match, and replay protection.

The Optimizer mints fresh authorizations both for forward calls (`_authorize_apply`) and rollbacks. Rollback always mints a separate authorization scoped to the *original* payload — so even if a malicious caller had captured the forward authorization, replaying it during rollback would fail (single-use + payload mismatch).

## Divergence from the directive

The directive specifies `approval_token: str | None`. The shipped implementation uses `ApprovedAction`. The differences:

| Property | Directive's `approval_token` | Shipped `ApprovedAction` |
|---|---|---|
| Presence check | yes | yes |
| Action binding | implicit | explicit (`action_name`) |
| Payload binding | none | SHA-256 hash of canonical args |
| Expiry | none | mandatory `expires_at`, validated at consume time |
| Single-use | none | `authorization_id` tracked in client; replay rejected |
| Tier binding | none | `approval_tier` cross-checked against `permissions.check()` |
| Audit provenance | none | optional `approved_by` / `source_envelope_id` / `source_snapshot_id` |

The threat model the directive's design covers is: "future caller forgets the gate." `ApprovedAction` covers that *and*: replay attacks (reuse a captured token), payload substitution (use the token to call a different action), expiry (stale tokens), and tier-confusion (claiming AUTO for a REQUIRES_APPROVAL action). For a system that may run autonomously and persist runs to disk (Phase 10), the stronger primitive is justified.

The directive's `permission_checker` is not injected into the constructor because the existing `permissions.check()` is a stateless module function and is consulted both at authorization *minting* (in `auto_authorize` / `human_authorize`) and at *consumption* (in `_tier_matches_permission_model` validator). The injection point would add nothing.
