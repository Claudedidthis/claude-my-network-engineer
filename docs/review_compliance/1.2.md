## Summary

`src/network_engineer/tools/ssl_policy.py` introduces a three-mode SSL
policy resolved from the `UNIFI_SSL_MODE` env var (default `lan_only`):

| Mode | verify= value | Construction-time check |
|---|---|---|
| `lan_only` | `False` | host MUST be RFC1918 / loopback / link-local IP literal — refuses public hosts and refuses hostnames (no resolution to avoid TOCTOU) |
| `ca_bundle` | path to `UNIFI_CA_BUNDLE` PEM | env var must be set, file must exist |
| `pinned` | (verifier deferred) | `UNIFI_CERT_FINGERPRINT` must be set; raises `NotImplementedError("Phase 14")` until the fingerprint verifier ships with the Cloudflare Tunnel deployment |

## Why three modes

Single `verify=False` conflates two threat models. On a LAN the realistic attacker has already compromised the LAN — cert verification isn't the operative defence. Off-LAN (Cloudflare Tunnel exposes the controller to the public internet), `verify=False` aligns autonomous actuation with a MITM attacker's capabilities — Waiter §5.3 names this an architectural resonance condition.

The directive specifies the configuration posture must be correct *before* the deployment topology changes — which is exactly now, while still LAN-only, but knowing Phase 14 is on the roadmap. Operators staging the env for tunnel deployment can set `UNIFI_SSL_MODE=pinned` plus `UNIFI_CERT_FINGERPRINT` and have the env validated immediately; the verifier itself ships with Phase 14.

## PINNED stub rationale

`httpx` does not natively support fingerprint pinning. Implementing it requires either a custom `ssl.SSLContext` with a verify callback, or an `httpx.HTTPTransport` subclass with cert-chain inspection. That is real Phase-14 work and not blocking the current LAN-only deployment.

The shipped behaviour: `UNIFI_SSL_MODE=pinned` validates the env (refuses missing fingerprint) and then raises `NotImplementedError` with a clear message pointing to Phase 14. Operators can stage the env without surprises; the verifier lands when it's actually load-bearing. The compliance status is `complete (with PINNED stub)` — the directive's three explicit acceptance items all pass; the deferred work is captured as a Phase 14 dependency.

## Wiring point

`UnifiClient.__init__` calls `resolve_mode()` and `build_verify(mode, host)` once at construction. Both `httpx.Client` instances (Network API + Protect API) receive the resulting `verify=` value. The v1 Integration API client (`unifi-sm-api`) accepts only a boolean `verify_ssl` flag, so it degrades to `verify_arg is not False` — when `CA_BUNDLE` is in use, the operator must add the CA to the OS trust store for the v1 transport. Documented in `.env.example`.

## Hostname handling

`_is_private_address` does NOT resolve hostnames. A hostname under `LAN_ONLY` is refused at construction. Rationale: resolving at construction creates a TOCTOU window (the resolution at construction is not the resolution at request time), and the documented LAN deployment uses the IP literal anyway. Operators using DNS for the controller should run `CA_BUNDLE` mode with a properly-signed cert.

## Backwards compatibility

The previous default behaviour was `verify=False` against any host. The new default is `verify=False` for private hosts (same effective behaviour for the documented home-LAN shape) but **fails fast** for misconfigured public hosts. An operator who today has `UNIFI_HOST` pointing at a public address (uncommon, almost certainly a misconfiguration) will see a clear error on next start instead of silently disabled cert verification — this is the desired posture change.

The deprecated `UNIFI_VERIFY_SSL=false` env var is no longer read; `.env.example` removes it and replaces it with the `UNIFI_SSL_MODE` block. Migration is one-liner: `UNIFI_VERIFY_SSL=false` → `UNIFI_SSL_MODE=lan_only` (the new default, so deletion of the old line also works).
