## Summary

Three additions to the dismissals registry:

1. **Effective-mandatory `expires_at`** — schema field stays optional for backwards compatibility, but `DismissalRegistry.load()` assigns a 90-day default TTL relative to `created_at` for any entry missing one and emits a structured warning. Operators see "this entry needs an explicit expiry" without losing the suppression mid-session.

2. **Fingerprint-based auto-revocation** — three new `Dismissal` fields:
   - `target_fingerprint: str | None` — sha256 of stable target attributes captured at dismissal-creation time.
   - `target_fingerprint_alg: str = "sha256-v1"` — versioned so the algorithm can evolve.
   - `reconfirm_on_change: bool = False` — opt-in. When True, callers passing `live_target_attributes` to `matches()` cause the registry to recompute the fingerprint and auto-revoke if it diverges.

3. **Stale-dismissal surfacing** — `DismissalRegistry.stale_dismissals()` returns a `list[StaleDismissal]` with each currently-inactive entry and the reason (`expired` or `fingerprint_mismatch`). Designed for the auditor to emit one INFO-severity finding per stale dismissal so the operator is prompted to re-confirm rather than the suppression silently disappearing.

## Aliasing-defence motivation

Per directive 1.4 / Waiter §5.4: a dismissal is a claim about network state. State drifts. Without TTL + fingerprint reconfirmation, an indefinite dismissal aliases real findings — a camera replacement (new MAC, new firmware, same name) inherits the old dismissal and the suppression compounds against ground truth between checkpoints. The registry becomes increasingly out of sync with the network without ever surfacing the divergence.

The combination of TTL (forces periodic re-confirmation) + fingerprint (catches mid-cycle target swaps) closes both halves of the aliasing surface.

## Backwards compatibility

The schema-level change is **strictly additive**:

- All three new fields have safe defaults (`None`, `"sha256-v1"`, `False`).
- Legacy YAML without `expires_at` still loads; the loader warns + assigns a default TTL.
- The `matches()` API gains an optional `live_target_attributes` kwarg; existing callers (auditor, monitor) continue to work without it. Fingerprint reconfirmation is skipped when the kwarg is absent — the existing TTL-only behaviour is preserved.

## Outstanding

- **Auditor wiring** — `stale_dismissals()` returns the data; the auditor's `run()` should iterate that list and emit `INFO` findings with code `DISMISSAL_STALE` and the operator's reason + the offending field path. Small change against the existing finding-emission path.
- **Live-attribute supply at consume sites** — currently auditor / monitor call `matches(code, evidence)` without `live_target_attributes`. To activate fingerprint reconfirmation in production, those call sites need to pass the relevant subset of the snapshot. Not blocking — `reconfirm_on_change` defaults to False so existing dismissals continue to work as-is.
